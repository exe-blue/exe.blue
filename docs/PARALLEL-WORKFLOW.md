# Parallel Workflow

여러 AI 에이전트가 동시에 작업한다. 충돌 없이 병렬성을 극대화하는 규칙.

## 원칙

가장 많은 병렬성을 만들고 있는 것은 인공지능이다.
이 레포는 그 병렬성을 위한 현실(코드)과 도달점(완료 기준)을 제공한다.

## 모듈 경계

각 AI는 자기 모듈/디렉터리에서 작업한다. 경계가 겹치면 명시적 합의 후 진행 — PR 코멘트에서 합의.

표준 디렉터리 경계:
- `/apps/<product>` — 제품별 (doai-me, ruon-ai, curatedx, aix)
- `/papers/<project>` — 학문별 (rps, art, tct, pam, act, set, uct, qrt, mur)
- `/docs/` — 헌법적 문서 (변경에 ORIGIN 승인 필요)
- `/specs/` — 사양 (AI 자율 머지 가능)
- `/scripts/` — 운영 스크립트

## 자동 머지 조건

1. 1개 이상의 다른 AI 리뷰 LGTM
2. 테스트 통과 (있는 경우)
3. ORIGIN의 명시적 거부 없음 (24시간 무응답 = 묵시적 승인)

이 세 조건을 만족하면 리뷰 AI가 직접 머지 가능.

## 충돌 해결

| 충돌 유형 | 해결 방식 |
|---|---|
| 같은 파일 동시 작업 | 먼저 PR 올린 쪽 우선. 뒤늦은 PR은 rebase |
| 설계 충돌 | PR 코멘트에서 두 AI 직접 합의. 무합의 시 ORIGIN 거부권 |
| 우선순위 충돌 | TASKS.md 상위 우선순위 우선 |

## 작업 잡기 워크플로우

```
1. TASKS.md "다음 우선순위" 확인
2. 미할당 과제 선택
3. PR 생성 → TASKS.md "In Progress"에 자기 줄 추가
4. 자기 브랜치(<agent>/<linear-id>-<slug>) 생성
5. 작업 → 커밋 → PR 본문 작성
6. 리뷰 요청 (다른 AI)
7. LGTM 받으면 머지
8. TASKS.md에서 자기 줄 제거
9. Linear 이슈 Done 전환
10. 다음 과제로
```

## 도달점 (Definition of Done)

도달점 없는 작업은 시작하지 않는다.

| 작업 유형 | Done 기준 |
|---|---|
| 코드 기능 | 테스트 통과 + PR 머지 |
| 문서 | AI 리뷰 + PR 머지 |
| 설계 사양 | ORIGIN 거부권 행사 기간 경과 + PR 머지 |
| 학술 논문 | 초안 완성 + AI 리뷰 + ORIGIN 최종 승인 + Zenodo 출판 |

## 코멘트 에티켓

- ORIGIN에게 묻지 않는다. 다른 AI에게 묻는다.
- 한 AI가 응답하지 않으면 다른 AI에게 묻는다.
- 합의가 안 되면 ORIGIN에게 결정 요청 — 단 명확한 옵션 2-3개를 제시. ORIGIN이 새 옵션을 만들지 않게 한다.

## 측정

병렬성의 지표:

- 동시 진행 PR 수
- 머지된 PR / 일
- AI 간 리뷰 횟수
- ORIGIN 호출 빈도 (낮을수록 좋음)

목표: ORIGIN 호출 빈도 → 0 (불가능, 하지만 방향)
